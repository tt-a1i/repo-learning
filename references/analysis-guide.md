# 分析指引

给 Agent 用的短清单。不要把本文暴露给最终用户。

## 产品海拔

读者要带走的是：**项目价值、架构、脉络原理**。

| 层 | 做什么 | 不要做什么 |
|----|--------|------------|
| 调查 | 跟调用、对证据、可记 `path:行号` | 把细读笔记原样贴进页面 |
| 呈现 | 职责、因果、取舍、价值 | 行号、函数清单、目录树、依赖名录 |

合上页后，读者应能用自己的话说清：它为什么重要、系统怎么转、模块如何分工、一次主请求/命令如何走完。

## 先定仓库形态与脊柱

形态决定叙事，不要强行套成 Web App：

- 终端产品 / 应用
- 库或 SDK
- CLI / 开发者工具
- 服务或分布式系统
- 数据 / ML 流水线
- 基础设施 / 配置
- Monorepo（先讲包边界）

大仓 / monorepo：用户未指定范围时，默认只深讲**主产品路径**（真实用户/请求走过的脊柱）；其余包在架构图边缘一笔带过，并在「未知与边界」写明未覆盖范围。深度来自脊柱，不来自覆盖率。

## 调查顺序

1. 项目自我描述（README、docs）→ 当假说
2. 价值线索：目标用户、要解决的痛、文档中的对比/动机；再用源码入口与主路径核实
3. 清单文件（package.json、go.mod、pyproject、Cargo.toml、workspace）→ 边界
4. 可执行入口与公开 API；选定脊柱
5. 组合根（main、bootstrap、DI、router 注册）
6. 抽 1 条（最多 2 条）真实主脉络，顺着调用读下去
7. 状态、持久化、外部系统、队列——只关心影响架构取舍的部分
8. 测试里能证明意图的例子（可选）

关系与步骤在调查笔记里可锚定路径；写进页面时升维成职责与阶段。

## 页面叙事（自顶向下）

1. **一句话**：它解决谁的什么问题
2. **项目价值**：为谁、解决什么、相对常见替代方案强在哪（2～5 句，有依据；禁止空泛营销）
3. **心智模型**：系统如何转起来（3～6 句）+ **为何长成这样**（2～4 句取舍：边界、同步/异步、插件 vs 单体等）
4. **结构**：职责单元与关系（Mermaid flowchart，3～8 节点）
5. **动态**：一条主脉络（阶段级 flowchart 或 sequenceDiagram）
6. **语言**：仅「不懂就读不懂上图/主流程」的项目特有概念
7. **若要继续读码**（可选）：≤3 个入口 + 短理解顺序；整节能删就删
8. **边界**：还不知道什么、本页不覆盖什么

少而准。宁可缺章节，不要空章节。

### 项目价值写什么

回答三问即可，不要写成广告：

1. **谁**在什么情境下需要它？
2. **没有它**时，人通常怎么凑合（手工、另一类工具、更大系统里的边角）？
3. **源码所体现的设计**如何兑现这个优势（可与后文架构呼应一句）？

价值必须能被 README 主张 + 入口/主路径证据支撑；对不上就降级为「文档自称…，源码侧尚未证实」，或写入未知。

## 图表

默认 **Mermaid**（Agent 好写、版式稳定）。填进模板的 `{{ARCHITECTURE_MERMAID}}` / `{{FLOW_*_MERMAID}}`。

| 图种 | 何时用 |
|------|--------|
| `flowchart LR` / `TB` | 架构、模块关系、生命周期 |
| `sequenceDiagram` | client / api / worker 等多角色协作 |
| `flowchart` 子图 `subgraph` | 清晰分层时再用，别堆砌 |

示例（架构 — 职责级）：

```mermaid
flowchart LR
  ingest[Ingest 采集] -->|快照| graph[Graph 实体]
  graph -->|索引| retrieve[Retrieve 问答]
```

海拔规则：

- **架构节点** = 职责/子系统，不是目录名、不是每个 package、不是 `utils`
- **流程节点** = 一次真实旅程的阶段（约 5～9 步），不是函数调用链
- 边写短标签（请求、写入、调度、发布…），不写「调用了 xxx()」
- 节点用项目自己的名字；空泛 Frontend / Backend 仅当仓库就这么叫
- 图与正文说同一件事；图不是装饰
- 架构节点大约 3～8；流程步骤大约 5～9；太大就只画脊柱
- 不要手搓 SVG 坐标，除非 Mermaid 表达不了且用户明确要求

## 关键概念录取

只收会改变「如何读架构/主流程」的词。框架常识（React、gRPC、middleware…）除非被赋予项目特有含义，否则不收。

每条：本项目里指什么 + 它卡住哪条边界/哪一步。

## 文案语气

冷静、精确、可亲近。像一位靠谱同事在白板上讲，而不是营销稿或审计报告。

## 页面气质（对齐模板）

浅色编辑部研报：纸色底 `#FBFAF7` + 青绿强调 `#0E6E6E`，Source Serif / Inter，sticky 顶栏用短章节名锚点（价值 / 心智模型 / 架构 / …），masthead + KPI 栅格，章节 `sec-head`，Mermaid 嵌在浅底 panel。不要退回暗色霓虹图鉴，也不要发灰的「说明书默认皮」。

KPI（形态 / 入口 / 置信度 / 范围）填不上就删掉整块 `.stats`，不要留空壳。

## 理解型验收（自检）

- 能否一句话说明价值？
- 能否指着架构图讲清模块分工？
- 能否顺着主流程讲完一次完整因果？
- 是否避免了文件树 / 依赖列表 / 调用链冒充理解？
